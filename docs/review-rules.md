# Devfolio Blog Review Rules

## 目标与适用范围

这份规则用于 `devfolio-blog` 仓库中的代码评审，适用于：

- 人工 PR review
- AI / agent review
- 前端、后端、共享库、内容数据类改动

这份规则只服务当前仓库，不是通用模板。评审时默认结合本仓库的 `Nx + Angular + NestJS + shared libs` 结构进行判断。

## Review 输出要求

评审输出必须采用 `Findings first` 风格：

1. 先列问题，再给总结。
2. 每条问题都要包含：
   - 严重级别：`P0` / `P1` / `P2` / `P3`
   - 问题描述
   - 影响范围或回归风险
   - 建议修复方向
3. 如果没有发现问题，必须明确写出 `No findings`。
4. 总结部分只做简短结论，不要把“改了什么”当成主要内容。

建议输出格式：

```text
Findings
- [P1] 问题标题
  问题：...
  影响：...
  建议：...

Open questions / Assumptions
- ...

Summary
- ...
```

如果没有问题：

```text
Findings
- No findings

Residual risks
- ...
```

## 必须检查项

每次 review 都必须检查以下内容：

- 功能是否正确，是否和需求或现有行为一致。
- 是否引入明显回归风险，尤其是已有页面、接口、内容路由和多语言能力。
- 是否补上了必要的测试、构建验证或至少说明了未验证项。
- 是否破坏现有模块边界，尤其是 `site`、`admin`、`api` 与 `libs/*` 之间的职责。
- 是否出现重复实现、硬编码、无边界扩散的逻辑或数据。
- 是否存在死链、错误资源路径、未同步的 locale 文案、无效图片引用。
- 是否保持可维护性，包括命名清晰、结构可读、改动范围合理。

## 按层次的专项检查

### 前端：Angular / Site / Admin

- 路由改动是否与当前信息架构一致，是否影响 `/:locale/*` 路径约定。
- 页面改动是否影响 SSR、构建结果或开发热更新行为。
- 组件是否复用已有模式，而不是重复堆模板和样式。
- 是否兼顾桌面端和移动端展示，避免明显换行、溢出、空白区域或布局断裂。
- 是否考虑可访问性：按钮语义、`aria-label`、图片 `alt`、可点击区域清晰度。
- 状态是否收敛在合适层级，是否避免页面组件里无边界堆积逻辑。

### 后端：NestJS / API

- 接口契约是否清晰，DTO、实体、返回结构是否一致。
- 是否正确处理错误路径、空数据、非法输入和未发布数据访问。
- 模块边界是否清晰，避免 controller/service/entity 之间职责混乱。
- 数据改动是否考虑持久化风险、兼容性和默认值行为。
- 新增接口是否至少具备最基本的可测试性和可验证性。

### 共享层：`libs/*`

- `libs/shared-types` 变更后，前后端和内容层是否同步。
- `libs/content-data` 修改后，文案、结构和页面使用方是否匹配。
- `libs/i18n` 新增字段后，中英文是否成对补齐。
- `libs/ui` 或共享视图逻辑改动后，是否影响多个页面而未做回归检查。

### 内容与资产

- 中英文内容是否对齐，避免只改中文或只改英文。
- 图片、PDF、静态资源路径是否真实存在，命名是否稳定可维护。
- 是否产生重复文案、重复卡片信息或结构性冗余。
- 内容型改动是否影响页面可读性，而不是只“把信息塞进去”。

## 常见直接拒绝项

出现以下情况时，默认应给出明确 finding，而不是轻描淡写：

- 改动后无法构建，或明显破坏现有启动方式。
- 新增页面或组件却没有考虑多语言结构。
- 修改共享类型或内容模型，但使用方未同步更新。
- 路由、图片、文档链接存在死链或错误路径。
- 只修局部显示问题，却引入更大范围的结构性重复或样式分叉。
- 为了快速实现而绕过已有模块边界，把逻辑硬塞进页面或单文件中。
- 提交中包含与当前目标无关的顺手修改，但没有说明原因。

## Review 结论模板

### 有问题时

```text
Findings
- [P1] 这里写最重要的问题
  问题：说明实际缺陷或回归点。
  影响：说明会影响哪些页面、接口、用户路径或维护成本。
  建议：给出明确修复方向，不写空泛建议。

- [P2] 第二个问题
  问题：...
  影响：...
  建议：...

Summary
- 这次改动的主要风险在于 ...
```

### 无问题时

```text
Findings
- No findings

Residual risks
- 已检查构建与关键路径，但仍建议在 ... 场景下做一次人工回归。

Summary
- 改动与当前仓库结构一致，未发现明显功能或结构性风险。
```
